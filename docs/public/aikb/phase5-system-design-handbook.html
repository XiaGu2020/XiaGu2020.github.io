1.1 五句话原理

第 2 章:Model Gateway / 多模型路由

2.1 五句话原理

① Model Gateway 是 LLM 系统的「Spring Cloud Gateway」:统一接入层,屏蔽下游模型差异,做路由、缓存、限流、计费、可观测。
② 五大职责:① 协议适配(OpenAI 协议 → 各模型协议)② 路由(按难度/租户/SLA)③ 多级缓存 ④ 限流熔断 ⑤ 计费可观测。
③ 为什么必须有:没有 Gateway 业务直连模型 = 业务被绑死 + 升级成本指数级 + 无法成本治理 + 无法做 A/B。

⑤ 主流方案:自研(大厂主流,深度定制)、LiteLLM(轻量开源)、Portkey(SaaS)、Cloudflare AI Gateway(边缘部署)。

2.3 完整代码:Gateway 核心调度

class ModelGateway:
    def __init__(self):
        self.router = ModelRouter()
        self.cache = MultiLevelCache()    # Result + Semantic
        self.limiter = TokenBucket()
        self.circuit = CircuitBreaker()
        self.tracer = Tracer()

    async def chat(self, req):
        with self.tracer.span("gateway.chat", user=req.user_id) as span:
            # ① 限流
            if not self.limiter.allow(req.user_id, req.estimated_tokens):
                raise TooManyRequests()
            # ② 多级缓存
            if hit := await self.cache.get(req):
                span.set("cache_hit", True)
                return hit
            # ③ 路由选模型
            model = self.router.route(req)
            span.set("model", model)
            # ④ 上游调用 + 熔断 + Fallback
            try:
                if self.circuit.is_open(model):
                    model = self.router.fallback(model)
                resp = await call_upstream(model, req)
            except UpstreamError as e:
                self.circuit.record_fail(model)
                resp = await call_upstream(self.router.fallback(model), req)
            # ⑤ 写缓存 + 记账
            await self.cache.set(req, resp)
            self.billing.charge(req.user_id, model, resp.usage)
            return resp

第 3 章:语义缓存 + 限流 + 熔断

3.1 五句话原理

① 三级缓存命中率叠加:Result Cache(精确)→ Semantic Cache(语义)→ Prefix Cache(KV)。典型审核场景三级叠加 50%+ 命中。
② Semantic Cache 关键:用 embedding 算相似度 ≥0.95 视为命中,但要防误命中——必须看场景温度阈值(医疗/支付场景阈值要 0.99)。
③ 限流三层:用户级(QPS+TPM)、租户级(配额)、全局级(熔断保上游)。LLM 必须按 token 限流不能只按 QPS,因为单请求 token 差异巨大。
④ 熔断必须细粒度:到「模型 × 实例」级别,不要整个上游一刀切。Half-Open 试探恢复。
⑤ 优雅降级:大模型不可用降级小模型;LLM 不可用降级规则;实在不行返回 Apology。永远不能让用户看到 500。

3.2 Semantic Cache 实现

class SemanticCache:
    \"\"\"基于向量相似度的语义缓存\"\"\"
    def __init__(self, threshold=0.95, ttl=3600):
        self.vec_store = Milvus(dim=1024)   # 存请求 embedding
        self.kv = Redis()                    # 存对应响应
        self.threshold = threshold
        self.ttl = ttl

    async def get(self, request):
        # ① 先查精确命中(prompt hash)
        if hit := await self.kv.get(f"exact:{hash(request.prompt)}"):
            return hit
        # ② 语义近似搜索
        emb = await embed(request.prompt)
        results = self.vec_store.search(emb, top_k=1)
        if results and results[0].score >= self.threshold:
            # 二次校验:输入参数(如 temperature/system_prompt)必须一致
            cached = await self.kv.get(f"vec:{results[0].id}")
            if cached and same_params(cached, request):
                return cached.response
        return None

    async def set(self, request, response):
        emb = await embed(request.prompt)
        vid = await self.vec_store.insert(emb)
        await self.kv.setex(f"vec:{vid}",
                            json.dumps({"req": request, "response": response}),
                            ex=self.ttl)
        await self.kv.setex(f"exact:{hash(request.prompt)}",
                            response, ex=self.ttl)

第 4 章:Golden Set + 评估流水线

4.1 五句话原理

① Golden Set 是 LLM 系统的「单元测试集 + 回归测试集」:200-1000 条精选 case,覆盖正常 / 边界 / 对抗 / 业务关键场景,有标准答案。
② 评估流水线四件套:Golden Set 跑分(回归) + LLM-as-Judge(主观打分) + 业务任务指标(准确率/采纳率) + 用户反馈(线上回流)。
③ 关键认知:不要只看 perplexity,perplexity 跟业务任务相关性弱——必须建业务任务套件。
④ Golden Set 维护是产品工作不是技术工作:需求 PM、业务专家、审核员一起参与;每月迭代;失败 case 必须分类沉淀。
⑤ 评估流水线必须自动化:每次模型/Prompt/RAG 配置变更自动跑 Golden Set,通过率达标才允许发布——这是 LLMOps 的根基。

    def run(self, target_system):
        results = []
        for case in self.golden:
            actual = target_system(case["input"])
            score_obj = self.score(case, actual)
            results.append({**case, "actual": actual, **score_obj})

        # 维度聚合
        by_cat = defaultdict(list)
        for r in results:
            by_cat[r["category"]].append(r["pass"])
        report = {
            "overall_pass_rate": sum(r["pass"] for r in results)/len(results),
            "by_category": {k: sum(v)/len(v) for k,v in by_cat.items()},
            "critical_pass_rate": self._critical(results),
            "fail_cases": [r for r in results if not r["pass"]],
        }
        return report

    def score(self, case, actual):
        # 多策略打分
        if case.get("type") == "exact":
            p = case["expected"] == actual
        elif case.get("type") == "json_schema":
            p = validate_json(actual, case["schema"])
        elif case.get("type") == "llm_judge":
            p = llm_judge(case["input"], case["expected"], actual) > 0.7
        elif case.get("type") == "rouge":
            p = rouge_l(case["expected"], actual) > 0.6
        return {"pass": p}

    def _critical(self, results):
        crit = [r for r in results if r["category"] == "critical"]
        return sum(r["pass"] for r in crit) / max(len(crit),1)

            return cfg["new"]
        return cfg["old"]

5.4 A/B 显著性检验

显著性需:n ≥ 16 / (delta² × p × (1-p)),delta 是期望提升幅度

例:基线采纳率 35%、希望识别 +3% 提升 → n ≥ 16 / (0.03² × 0.35 × 0.65) ≈ 7800 条。低于这个量级不要轻易下结论。
