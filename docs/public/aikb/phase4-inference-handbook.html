第 1 章:Prefill vs Decode——LLM 推理的两阶段

1.1 五句话原理

① LLM 推理分两阶段:Prefill(预填充) + Decode(解码)。Prefill 一次性处理 prompt 所有 token 算出第一个 token + 全部 KV;Decode 逐 token 自回归生成。
② Prefill 是计算密集型(compute-bound): 一次性算 N 个 token 的全部 attention,GPU FLOPS 拉满,显存带宽不是瓶颈。
③ Decode 是显存带宽密集型(memory-bound):每步只生成 1 个 token,但要读全部权重 + KV Cache,FLOPS 利用率往往 <10%。
④ 这两阶段计算特征完全不同,所以优化手段也不同:Prefill 优化算子(FlashAttention),Decode 优化访存(KV Cache + Continuous Batching + 投机解码)。
⑤ TTFT 由 Prefill 决定,TPOT 由 Decode 决定——分开优化才能拿满。面试官最爱问「为什么 Decode 慢」,答「memory-bound,瓶颈在 HBM 带宽,不在算力」就稳。

1.2 两阶段对照表

瓶颈FLOPS(算力)HBM 带宽(显存)
FLOPS 利用率50-90%5-15%
对应指标TTFT(首 token 延迟)TPOT(每 token 延迟)
优化方向FlashAttention / Tensor CoreKV Cache / 批处理 / 投机解码

1.3 完整代码:Profile 两阶段耗时

import time, torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen3-4B",
                  torch_dtype=torch.float16, device_map="cuda")
tok = AutoTokenizer.from_pretrained("Qwen/Qwen3-4B")

prompt = "审核以下章节内容是否合规:" + "..." * 500   # 长 prompt
inputs = tok(prompt, return_tensors="pt").to("cuda")

# ① Prefill:一次前向算所有 prompt token + 第一个新 token
torch.cuda.synchronize(); t0 = time.time()
with torch.no_grad():
    out = model(**inputs, use_cache=True)
    past_kv = out.past_key_values
    next_tok = out.logits[:,-1,:].argmax(-1, keepdim=True)
torch.cuda.synchronize()
prefill_ms = (time.time()-t0)*1000
print(f"Prefill {inputs.input_ids.shape[1]} tokens: {prefill_ms:.1f}ms")

# ② Decode:逐 token 生成,每步只输入 1 token
gen, t1 = [next_tok], time.time()
for _ in range(99):
    with torch.no_grad():
        out = model(input_ids=gen[-1], past_key_values=past_kv, use_cache=True)
        past_kv = out.past_key_values
        gen.append(out.logits[:,-1,:].argmax(-1, keepdim=True))
torch.cuda.synchronize()
decode_ms = (time.time()-t1)*1000
print(f"Decode 100 tokens: {decode_ms:.1f}ms, TPOT={decode_ms/100:.1f}ms/tok")

1.4 易踩坑

坑 1:用 Prefill 的延迟评估整体性能。大量场景 Decode 才是大头(生成 500 token 时 Decode 占 90% 时间)——只看 TTFT 会误判。
坑 2:把 Prefill 当成可以批的随便批。Prefill 已经把算力占满了,叠太多请求反而互相挤算力。Decode 才是批处理收益最大的阶段。
坑 3:不用 KV Cache 重算每步。不传 past_key_values 每步都重算前面所有 token——TPOT 直接爆 10 倍。
坑 4:误以为 Decode 慢是模型大。Decode 慢的根本原因是访存量大(权重 + KV)不是算力,模型再小如果 KV 没压缩 Decode 还是慢。
坑 5:Prefill 阶段不做 chunk。32K 长 prompt 一次 Prefill 显存爆——vLLM 的 chunked prefill 把它切成小块串行算。

第 2 章:TTFT / TPOT / 吞吐——推理三大核心指标

2.1 五句话原理

① 推理服务三个根指标:TTFT、TPOT、Throughput。面试讲不清这三个直接挂掉。
② TTFT(Time To First Token):从请求到达起,到生成第一个 token 的延迟。由 Prefill 决定,影响用户感知的「响应快不快」。
③ TPOT(Time Per Output Token):生成相邻两个 token 的间隔。由 Decode 决定,影响「打字流畅度」。注意是 TPOT 不是 TBT(那是过时的叫法)。
④ Throughput(吞吐):单位时间整个服务处理的总 token 数(prefill_tokens + decode_tokens)/s。是成本的反向指标——吞吐越高单 token 成本越低。
⑤ 三个指标互相 trade-off:批越大吞吐越高但 TTFT/TPOT 越长;批越小延迟越好但吞吐越低。SLA 决定怎么权衡。

2.2 三指标对照 + 经验值

指标定义影响典型 SLA
TTFT请求 → 第一个 token用户感知响应快慢对话 ≤500ms;批处理 ≤2s
TPOTtoken 间间隔打字流畅度(应 ≥ 阅读速度)≤50ms(20 tok/s) 流畅
Throughput总 tokens/s/服务成本反向指标越高越好,7B 单卡 ≥3K
P99 端到端延迟TTFT + TPOT × NSLA 兜底对话 ≤5s;长输出 ≤30s

2.3 成本公式

单 token 成本 = GPU 小时单价 / Throughput(tok/s) / 3600

例:H100 单价 ¥30/h,服务跑出 5000 tok/s → ¥30/5000/3600 = ¥0.00000167 /tok ≈ ¥1.67 /M tok。这就是为什么吞吐是成本反向指标。

2.4 完整代码:压测 vLLM 服务

# benchmark_serving.py(简化版)
import asyncio, time, aiohttp, json, statistics

async def one_request(session, prompt, n_out=200):
    t0 = time.time()
    first_token_t = None
    async with session.post("http://localhost:8000/v1/completions",
        json={"model":"qwen3-4b","prompt":prompt,"max_tokens":n_out,"stream":True}) as resp:
        n_tokens = 0
        async for line in resp.content:
            if line.startswith(b"data:") and line.strip()!=b"data: [DONE]":
                if first_token_t is None:
                    first_token_t = time.time()
                n_tokens += 1
        end_t = time.time()
    ttft = (first_token_t - t0)*1000 if first_token_t else 0
    tpot = (end_t - first_token_t)*1000 / max(n_tokens,1)
    return ttft, tpot, n_tokens, end_t-t0

async def main(concurrency=32, total=1000):
    async with aiohttp.ClientSession() as session:
        sem = asyncio.Semaphore(concurrency)
        async def run(p):
            async with sem:
                return await one_request(session, p)
        results = await asyncio.gather(*[run("审核以下内容...") for _ in range(total)])

    ttfts = [r[0] for r in results]
    tpots = [r[1] for r in results]
    total_t = max(r[3] for r in results)
    total_tok = sum(r[2] for r in results)
    print(f"TTFT P50={statistics.median(ttfts):.0f}ms P99={sorted(ttfts)[int(0.99*len(ttfts))]:.0f}ms")
    print(f"TPOT P50={statistics.median(tpots):.1f}ms P99={sorted(tpots)[int(0.99*len(tpots))]:.1f}ms")
    print(f"Throughput: {total_tok/total_t:.0f} tok/s")

asyncio.run(main())

第 3 章:KV Cache 原理与显存账

3.1 五句话原理

① KV Cache 是 Decode 加速的根基:注意力计算需要每个 token 的 K、V,前面的 token 不变,所以缓存起来下次复用,不重算。
② 没 KV Cache:每生成 1 个 token 都要重算前面 N 个 token 的 KV(O(N²) 复杂度爆炸)。
③ 有 KV Cache:每步只算新 token 的 K/V,追加到缓存,O(N) 复杂度。
④ 代价:KV Cache 占显存非常大。公式:KV = 2 × layers × heads × head_dim × seq_len × bytes。Llama-7B 一个 4K 序列 KV ≈ 2GB。
⑤ KV 大小决定单卡能塞多少并发请求——这是 PagedAttention 出现的根因。

3.2 KV Cache 显存计算公式

KV_size = 2 × n_layers × n_kv_heads × head_dim × seq_len × dtype_bytes × batch
模型layerskv_headshead_dim4K seq FP16 单序列
Llama2-7B3232(MHA)1282 × 32 × 32 × 128 × 4096 × 2 = 2.0 GB
Llama2-70B808(GQA)1282 × 80 × 8 × 128 × 4096 × 2 = 1.25 GB
Qwen3-4B368(GQA)1282 × 36 × 8 × 128 × 4096 × 2 = 0.56 GB
DeepSeek-V3611(MLA)576≈ 0.27 GB(MLA 大幅压缩)
三大节省 KV 的架构演进:
① MHA(Multi-Head Attention):每头独立 KV → 显存爆炸。
② GQA(Grouped Query):多个 Q 共享一组 KV → 节省 4-8 倍。Llama2-70B / Qwen3 都用。
③ MLA(Multi-Head Latent Attention):KV 压缩到低秩潜空间 → 节省 10x+。DeepSeek-V2/V3 独家。

3.3 完整代码:KV Cache 算账工具

def kv_size_gb(layers, kv_heads, head_dim, seq_len, batch=1, dtype_bytes=2):
    """返回 KV Cache 占用 GB"""
    bytes_total = 2 * layers * kv_heads * head_dim * seq_len * batch * dtype_bytes
    return bytes_total / (1024**3)

# Llama2-7B 4K 上下文,32 并发
print(kv_size_gb(32, 32, 128, 4096, 32))   # 64.0 GB → A100 80G 也吃力!

# Qwen3-4B 4K,32 并发
print(kv_size_gb(36, 8, 128, 4096, 32))    # 18.0 GB → 友好

# 单卡能跑多少并发?
def max_batch(gpu_mem_gb, model_weight_gb, layers, kv_heads, head_dim, seq_len, dtype_bytes=2):
    free = gpu_mem_gb - model_weight_gb - 2  # 留 2GB 给激活/buffer
    per_seq_gb = kv_size_gb(layers, kv_heads, head_dim, seq_len, 1, dtype_bytes)
    return int(free / per_seq_gb)

# A100 80G + Llama2-7B(14GB FP16)+ 4K
print(max_batch(80, 14, 32, 32, 128, 4096))  # 32 并发(理论上限)

3.4 易踩坑

坑 1:忽略 KV 占显存,塞模型不留余地。「7B 模型才 14G,A100 80G 能塞 5 个」错!1 个模型 + 32 并发 KV = 64GB,根本塞不了 5 个。
坑 2:把 MHA / GQA / MLA 当一回事。同一参数量模型,MHA 和 MLA 单卡并发能差 10 倍,选模型必须看这个。
坑 3:KV 不复用 prefix。同一 system prompt 被复用 1000 次也每次重算 KV——开 prefix-caching 能省 30-70% Prefill。
坑 4:KV dtype 没量化。FP16 是默认,但 INT8 KV Cache 能再省一半显存(精度损失 <1%)。vLLM/SGLang 都支持。
坑 5:不监控 KV 利用率。vLLM 有 `gpu_cache_usage_perc` 指标,>90% 说明要扩容了,<30% 说明并发设太小浪费显存。

第 4 章:PagedAttention——vLLM 的核心创新

4.1 五句话原理

① PagedAttention 是 vLLM 2023 论文的核心创新,灵感来自操作系统的虚拟内存分页。
② 解决的问题:朴素 KV Cache 给每个序列预分配 max_seq_len 大小的连续显存块——但实际平均长度可能只有 max 的 30%,显存浪费 70%。
③ 核心思想:把 KV Cache 切成固定大小的 Block(默认 16 个 token 一块),序列按需申请 Block,不连续——像 OS 把进程地址空间分页映射到物理内存。
④ 收益:显存利用率从 30% 提到 96%+,有效并发提 4-10 倍。是 vLLM 单卡吞吐碾压原生 PyTorch 推理的核心原因。
⑤ 副产品:不同序列可以共享相同的 prompt 前缀 Block(就是 prefix caching),进一步省显存 + 加速 Prefill。

4.2 朴素 KV vs PagedAttention 对照

维度朴素 KVPagedAttention
显存分配预留 max_seq_len 连续按需分 Block(16 token/块)
显存利用率20-40%90-96%
外部碎片严重无(Block 大小一致)
共享 prefix无法共享Block 引用计数,完全共享
类比 OS静态分区分页虚拟内存

第 5 章:Continuous Batching——连续批处理

5.1 五句话原理

① 朴素 Static Batching(GPU 训练那套):一批请求一起进、一起算、一起出——慢的拖快的,GPU 利用率被「最长那个」拖累。
② Continuous Batching(也叫 In-flight Batching):每生成一个 token 就重组 batch——已结束的请求从批里踢出去,新请求立刻塞进来。
③ 类比:Static Batching 像「公交车」(必须等齐发车),Continuous 像「滚动地铁」(随时上下车)。
④ 收益:GPU 永远在算满批,吞吐 5-10 倍提升。是 vLLM/TGI/SGLang/TensorRT-LLM 的标配。
⑤ 关键设计:调度器(Scheduler)每个 step 决定哪些序列继续 Decode、哪些新序列进来 Prefill、显存够不够要不要驱逐。

5.2 Static vs Continuous 对照图

Static Batching(每批等齐):
时间→  | step1 | step2 | step3 | step4 |
ReqA   |  *  |  *  |  *  |  *  | (生成 4 个 token 完成)
ReqB   |  *  |  *  | END | --- | (2 个 token 完成,空等)
ReqC   |  *  | END | --- | --- | (1 个,空 3 步)
ReqD   |  *  |  *  |  *  | END |
GPU 利用率: 100% 75% 50% 25% ← 后期严重浪费

Continuous Batching(滚动地铁):
时间→  | step1   | step2   | step3   | step4   |
ReqA   |   *     |   *     |   *     |   *(end)|
ReqB   |   *     |   *(end)| ReqE *  | ReqE *  |  ← B 走 E 来
ReqC   |   *(end)| ReqF *  | ReqF *  | ReqF *  |  ← C 走 F 来
ReqD   |   *     |   *     |   *     |   *(end)|
GPU 利用率: 100% 100% 100% 100% ← 满载

5.3 Scheduler 伪代码

class ContinuousScheduler:
    def __init__(self, max_seqs=256, max_kv_blocks=4096):
        self.running = []          # 正在 Decode 的序列
        self.waiting = deque()     # 等待 Prefill 的序列
        self.max_seqs = max_seqs
        self.kv_pool = BlockPool(max_kv_blocks)

    def step(self):
        # ① 已完成的序列从 running 踢出,释放 KV
        finished = [s for s in self.running if s.done]
        for s in finished:
            self.running.remove(s)
            self.kv_pool.free(s.blocks)

        # ② 看 KV 池剩余,决定能否塞新请求(从 waiting 拉)
        while self.waiting and len(self.running) < self.max_seqs:
            seq = self.waiting[0]
            need_blocks = ceil(seq.prompt_len / 16)
            if self.kv_pool.can_alloc(need_blocks):
                seq.blocks = self.kv_pool.alloc(need_blocks)
                self.running.append(seq)
                self.waiting.popleft()
                seq.do_prefill()    # 第一次进来要算 Prefill
            else:
                # ③ KV 不足:启动驱逐(swap to CPU / preempt)
                self.try_evict()
                break

        # ④ 所有 running 序列前进一步 Decode
        forward_pass(self.running)

        # ⑤ 检查需要追加 KV Block 的(序列变长)
        for s in self.running:
            if s.kv_full():
                s.blocks.append(self.kv_pool.alloc(1))

5.4 易踩坑

坑 1:Prefill 阶段不 chunk 阻塞调度。来一个 32K prompt 的请求,Prefill 一次 1 秒,期间所有 Decode 卡住——TPOT P99 飞升。开 chunked-prefill 把它切成 512 token 一块串行算。
坑 2:max-num-seqs 与 KV 不匹配。设 max-num-seqs=256 但 KV 池只够 64 序列 → 调度器频繁驱逐 swap → 抖动严重。要算账匹配。
坑 3:Preemption 策略没监控。vLLM 在 KV 紧张时会 preempt(把序列 swap 到 CPU 或重算),这是性能毒药。要监控 `num_preemption` 指标,大于 0 说明负载已到极限。
坑 4:不知道 Prefill / Decode 互相干扰。chunked-prefill + decode 同 step 跑,两者抢算力,TPOT 会被 Prefill 拖。SGLang/Mooncake 用 disaggregated 物理分离解决。
坑 5:Streaming 输出不会聚。Continuous Batching 下不同请求生成进度不同,客户端 streaming 必须按 request_id 分流,不能混在一起。

第 6 章:FlashAttention 与算子优化

6.1 五句话原理

① FlashAttention 是 2022 Tri Dao 的论文,把 Attention 算子从 O(N²) 显存压到 O(N),且更快。
② 朴素 Attention 问题:计算 softmax(QK^T) × V 需要 N × N 大小的中间矩阵 S 写到 HBM 又读回——大量 HBM 读写,memory-bound。
③ FlashAttention 核心:Tiling(分块)+ 在线 Softmax(online softmax),把 Q/K/V 切块到 SRAM 里算,中间结果不落盘 HBM。
④ 收益:速度 2-4x、显存 5-10x。FlashAttention-2 改进了并行度;FlashAttention-3 利用 H100 的 TMA + WGMMA。
⑤ 关键认知:2026 任何主流推理引擎都内置 FlashAttention,你不用自己写——但要能讲清原理,这是 P8 必考。

6.2 朴素 vs FlashAttention 数据流

朴素 Attention(naive):
HBM ←→ SRAM
1. 从 HBM 读 Q、K  → 算 S = QK^T → 写 HBM (N² 大小)
2. 从 HBM 读 S    → 算 P = softmax(S) → 写 HBM (N²)
3. 从 HBM 读 P、V → 算 O = PV  → 写 HBM
                  ↑ 大量 HBM 读写,memory-bound

FlashAttention:
1. 把 Q 切成 Q_blocks,K/V 切成 K_blocks/V_blocks
2. 每个 Q_block:
     for K_block, V_block in zip(K_blocks, V_blocks):
        加载到 SRAM
        在 SRAM 算 S_ij = Q_i K_j^T
        在 SRAM 用 online softmax 累加 → O_i
   只把最终 O_i 写回 HBM
                  ↑ S 和 P 永远不落盘 HBM,显存 O(N)、速度 2-4x

6.3 在线 Softmax 关键技巧

问题:Softmax 需要全局最大值减位防溢出,但 Tiling 时只能看到局部块。
解决:每块算完保存 (m_i, l_i)(局部最大值、局部分母和),下一块来时用修正因子合并:
m_new = max(m_i, m_j); l_new = exp(m_i - m_new) × l_i + exp(m_j - m_new) × l_j
这样跨块也能正确算出全局 softmax,不用回头看,O(N) 完成。

第 7 章:投机解码 Speculative Decoding

7.1 五句话原理

① 核心:用一个小模型(Draft)快速生成 N 个候选 token,大模型(Target)一次性验证——验证只要 1 次前向(Decode 是 memory-bound,N 个 token 一起验和验 1 个差不多)。
② 验证逻辑:把 Draft 生成的 N 个 token 拼到原序列,Target 一次前向算出每个位置的概率分布,逐位检查 Draft 选的 token 是否在 Target 的高概率区间——通过的接受,失败的截断重采。
③ 收益:1 次 Target 前向产出 K 个 token(K 取决于接受率),理论提速 ≈ 接受率 × Draft 模型速度差距。实际 1.5-3x。
④ 关键:接受率 + 小模型与大模型分布相似度。太相似但小模型慢→白做;太不相似接受率低→白做。
⑤ 进阶变体:Medusa(多 head 同时预测多 token)、EAGLE(用大模型 hidden state 训 draft,接受率高)、Lookahead(无需 draft 模型,基于 ngram 命中)。

第 9 章:LoRA / QLoRA 微调与服务化

9.1 五句话原理

① LoRA(Low-Rank Adaptation):不动原模型,只在每个 Linear 旁加两个低秩矩阵 A(d×r)、B(r×d)(r 通常 8-64),训练时只更新 A、B。
② 优势:① 训练参数减少 100-1000 倍;② 多任务可叠加(不同 LoRA 切换);③ 不破坏原模型能力。
③ QLoRA:把 base 模型量化到 INT4(NF4)冻结,LoRA 部分仍 FP16 训练——24G 显存训 70B 模型,平民化大模型微调的关键创新。
④ 服务化关键:vLLM/SGLang 都支持多 LoRA 同时挂载(LoRAX、Punica),一个 Base 模型挂 N 个 LoRA,不同请求路由到不同 LoRA——降本利器。
⑤ 关键认知:LoRA 不是万能。指令跟随、风格调整、领域知识对齐很合适;但学新世界知识、改基础推理能力效果差,要 SFT/CPT。

第 11 章:多卡并行(TP / PP / EP / DP)

11.1 五句话原理

① 单卡装不下 → 多卡。四种并行:Tensor Parallel(TP)、Pipeline Parallel(PP)、Expert Parallel(EP,MoE 专用)、Data Parallel(DP,多副本)。
② TP(张量并行):把每个 Linear/Attention 切到多卡上算,卡间同步(all-reduce)——延迟低,适合同节点高速 NVLink。
③ PP(流水线并行):把模型切成几段,每段一张卡——卡间通信少,但有 bubble(空闲气泡),适合跨节点 InfiniBand。
④ EP(专家并行):MoE 模型把不同专家放不同卡,只激活路由命中的专家——DeepSeek-V3、Mixtral 必备。
⑤ DP(数据并行):多个副本各自服务一部分流量,卡间不通信——是横向扩容的根基,但单实例延迟不变。

11.2 四种并行对照

并行切什么通信适用
TP每个 Linear 切 N 份每层一次 all-reduce同节点高速互联,7B-70B 单实例
PP模型分 N 段段间一次 P2P跨节点,超大模型
EPMoE 专家分卡路由 + all-to-allMoE 模型(DeepSeek/Mixtral)
DP多副本无横向扩吞吐

11.5 易踩坑

坑 1:跨节点用 TP。TP 每层 all-reduce,跨 IB 延迟翻倍——跨节点必须用 PP。
坑 2:TP 大于 head 数。TP=8 但 attention head 只有 4 → 切不动。Llama-3-8B head=32 最大 TP=32 (但实际 NVLink 8 卡为上限)。
坑 3:PP 不调 microbatch。PP 默认 microbatch=1 会有大量 bubble,要调 micro_batches=8/16。
坑 4:DP 没做 LoadBalancer 一致哈希。同 user 的请求路由到不同副本就丢了 prefix cache——LoadBalancer 按 session 哈希。
坑 5:不监控通信占比。TP/PP 通信能占 30%+,要看 nccl tracing 决定切多大才划算。

第 12 章:成本优化全景 + 推理演进路线

12.1 五句话原理

① 推理成本三大要素:GPU 单价、Throughput(tok/s/卡)、负载利用率(峰谷比)。
② 单 token 成本 = (GPU/h 单价) / Throughput / 3600。从 ¥30/M 降到 ¥3/M 的优化路径:模型选型 → 量化 → 并行 → 投机解码 → 缓存。
③ 缓存层是降本利器:Prefix Cache(KV 复用)、Semantic Cache(语义命中)、Result Cache(完全命中)三层。审核场景重复率高,缓存命中能降 60%+ 成本。
④ 模型路由(Model Gateway)是降本中枢:简单请求走小模型(0.5B),复杂请求走大模型(72B),按 difficulty 评分动态路由——平均成本下降 5-10x。
⑤ 演进三阶段:① 单卡 vLLM(MVP)→ ② 多卡 + 缓存 + 量化(成熟)→ ③ Disaggregated + Model Gateway + 弹性调度(规模化)。

12.3 Model Gateway 路由示例

class ModelGateway:
    \"\"\"按难度路由到不同模型,降本核心组件\"\"\"
    ROUTERS = [
        # (条件函数, 模型, 优先级)
        (lambda r: len(r.prompt) < 200 and not r.has_image, "qwen3-0.6b"),
        (lambda r: r.is_simple_classification, "qwen3-1.7b"),
        (lambda r: r.is_chinese_long, "qwen3-4b-awq"),
        (lambda r: r.is_complex_reasoning, "qwen3-32b"),
        (lambda r: r.has_image, "qwen3-vl-7b"),
    ]
    DEFAULT = "qwen3-4b"

    def route(self, request):
        # ① 缓存层:先查 result cache
        if hit := self.result_cache.get(hash(request)):
            return hit
        # ② 语义缓存(向量近似 0.95+ 命中)
        if hit := self.semantic_cache.search(request, threshold=0.95):
            return hit
        # ③ 难度路由
        model = self.DEFAULT
        for cond, m in self.ROUTERS:
            if cond(request):
                model = m
                break
        # ④ 调用 + 回写缓存
        result = call_vllm(model, request)
        self.result_cache.set(hash(request), result, ttl=3600)
        return result

12.5 易踩坑

坑 1:不算账盲优化。砸钱上 H100 但缓存没开 = 浪费。先把命中率拉到 50% 再考虑硬件升级。
坑 2:Cache 不分层。只做 Result Cache 命中率 5%,加 Semantic Cache 能拉到 40%。
坑 3:Gateway 没做路由可观测。不知道每个 Tier 命中多少,无法优化路由阈值。
坑 4:成本算法忽略 Cold Start。弹性扩缩容时 Cold Start 时间长,频繁起停反而贵——要算稳定保留+弹性溢出分层。
坑 5:把 SLA 当唯一指标。SLA 满足不等于最优,要做 Cost-SLA Pareto 曲线,看在保 SLA 前提下成本压到哪里是边际拐点。
