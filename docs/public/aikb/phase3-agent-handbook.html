1Agent 全景图:从 LLM 调用到 Agent 系统

1.1 五句话原理

  1. Agent = LLM 作为决策核心 + 一组工具(Tools)+ 一套循环(Loop)→ 让模型可以「看观察、做判断、调工具、再看结果、再判断」直到任务完成。
  2. LLM 单次调用是静态映射(input→output),Agent 是动态决策图(每一步基于上一步结果决定下一步)。
  3. 面试官最爱用一句话区分:「Chatbot 是回答问题,Agent 是完成任务」——任务往往需要多步、多工具、状态保持。
  4. Agent 五件套:LLM(脑) + Prompt(规则)+ Tools(手)+ Memory(记忆) + Observability(可观测),缺一不可。
  5. 2026 行业共识:Agent 不是越自由越好——越是关键业务,越要用 Workflow 约束 Agent,只在确定决策点放它自由。

1.2 LLM 调用 vs Agent 三种范式

形态核心典型场景稳定性
单次 LLM无状态、无工具翻译、改写、分类★★★★★
RAG固定一次「检索→生成」知识问答、规则查询★★★★
Workflow预定义 DAG,LLM 只在节点内决策审核流水线、订单处理★★★★
Agent(ReAct)LLM 自主选择下一步研究助手、复杂诊断★★ ~ ★★★
Multi-Agent多 Agent 协作 + 主管软件开发、深度研究★ ~ ★★
P8/P9 视角:「Agent」不是技术决定,是业务约束决定。能用 Workflow 解决的场景就别上 Agent——Workflow 可预测、可观测、可回滚;Agent 自由意味着不可预测。审核场景核心链路 99% 都该是 Workflow,只在「难例分诊、新规则起草」这种探索性环节用 Agent。

1.3 易踩坑

  1. 无脑用 Agent 替代 Workflow——业务关键流强行 Agent 化,线上不稳、Bad Case 排查难。
  2. 把 Agent 当 Demo 写——LangChain 教程级别的 Agent 上线,3 天爆 100 个工单。
  3. 不画状态机——Agent 跑哪一步、状态怎么变,讲不清就不能上线。
  4. 不留人工兜底——Agent 最大风险不是出错,是错得有自信。关键决策点必须有人工介入路径。

2ReAct vs Plan-Execute · 两大范式根因对比

2.1 五句话原理

  1. ReAct(Reasoning + Acting):每一步循环 = Thought → Action → Observation → Thought...,边想边做,LLM 每步都有最新信息。
  2. Plan-Execute:先一次性生成完整 Plan(N 步),再按计划逐步 Execute,想完再做。失败可触发 Re-plan。
  3. ReAct 优势:灵活、可应对意外;劣势:容易死循环、token 消耗大、轨迹不可预测。
  4. Plan-Execute 优势:可预测、可中断、可回放;劣势:Plan 错了一步全错、面对环境变化不灵活。
  5. 2026 工业实践:关键链路用 Plan-Execute(可控)+ 在子任务节点用 ReAct(灵活),这是混合范式。

2.2 范式对比图

ReAct 循环: Plan-Execute: Thought 1 ┌────────────┐ ↓ │ ① Plan │ LLM 一次性出 N 步 Action 1 ─→ Tool │ step1... │ ↓ │ step2... │ Observation 1 │ step3... │ ↓ └─────┬──────┘ Thought 2 ↓ ↓ ② Execute step1 → tool Action 2 ─→ Tool ↓ ↓ ③ Execute step2 → tool Observation 2 ↓ ↓ ④ 失败? → Re-plan ... 直到 Final Answer ↓ ⑤ Final Answer 灵活但易抖动 可预测但需要 Re-plan

2.3 决策表

维度ReActPlan-Execute
响应延迟累积式增长Plan 阶段慢 + Execute 快
token 消耗高(每步都带历史)中(Plan 一次,Execute 短)
失败处理下一轮自动调整需 Re-plan,可能整批废
可观测性差(轨迹动态)好(Plan 是可见 DAG)
适用探索型、对话型固定流程、批处理

    return {"plan": plan, "step": 0}

  1. ReAct 跑 30 步还没收敛——LLM 抖动 Thought 来回反复。要加最大步数(典型 10)、同 Action 重复检测(连续两次相同 tool+args 就强制结束)。
  2. Plan-Execute 失败一步整批废——要支持局部 Re-plan:仅重新规划失败步及后续,前面成功步保留。
  3. Plan 出来 20 步——LLM 喜欢列长 Plan 显得详细。要在 prompt 限制 max_steps=5,简单任务必要时强制 1 步。
  4. 把 Plan 当字面意思执行——Plan 里的「检查用户余额」其实是「调 get_balance 工具」。Plan 步骤要带 tool 和 args 的结构化字段,不能裸文字。

3Function Calling 工程化:从 demo 到生产

3.1 五句话原理

  1. Function Calling = LLM 输出结构化的工具调用意图(JSON: {name, arguments}),由你的运行时去真正执行,再把结果回灌给 LLM。
  2. Demo 级 Function Calling:1-2 个工具、参数简单、不考虑失败 → 上线 3 天必爆。
  3. 生产级要解决五件事:① Tool Schema 规范(JSON Schema 严格校验);② 参数注入安全(防止 SQL 注入、命令注入);③ 幂等与重试(同一调用不能扣两次款);④ 超时与熔断(单工具不能拖垮整链路);⑤ 结果回灌格式(给 LLM 看的不是裸输出)。
  4. 2026 主流模型(GPT-5、Qwen3、Claude 4.5、DeepSeek-V3)都原生支持 Function Calling,但跨模型协议有细微差异(OpenAI tools 数组 vs Anthropic tools+tool_use 块,参数名 strict mode 不同)——这是模型网关存在的理由。
  5. 审核场景里 Function Calling 最常见的工具:查规则、查历史、查用户画像、上报系统、给运营发通知——每个都要做「权限+审计+重试」三件套。

3.2 工具描述的「三个等级」

等级描述质量调用准确率
★(差)「query: 查询」~50%
★★★(普通)「query 字符串,搜索关键词」~80%
★★★★★(生产)「query 字符串,长度 1-100,使用业务名词而非 SQL,例:'用户余额'而非 'SELECT balance'。日期用 ISO 格式」+ 一两个 example>95%
P8 经验:Tool 的 description 比 LLM 的 Prompt 更重要。一个 Tool 描述写好,准确率从 70% 飙到 95%。每个工具都要写「干什么/参数边界/坑/示例」。这是Tool Engineering,2026 新名词。

3.3 代码:生产级 Tool 定义

from pydantic import BaseModel, Field
from typing import Literal

class CheckRuleArgs(BaseModel):
    rule_keyword: str = Field(..., description="审核规则关键词,使用业务名词,如'自伤''擦边',不要用 SQL。长度 2-30。", min_length=2, max_length=30)
    rule_version: Literal["V2.5", "V2.6"] = Field(default="V2.6", description="规则版本号,默认 V2.6")

def check_rule(args: CheckRuleArgs) -> dict:
    # 1. 权限校验
    if not ctx.user.has_permission("check_rule"): raise PermissionError
    # 2. 参数清洗(已被 pydantic 校验过类型/长度)
    # 3. 调用真实业务
    try:
        result = rule_service.search(args.rule_keyword, args.rule_version, timeout=3)
    except TimeoutError:
        return {"error": "timeout", "hint": "规则服务超时,请稍后重试或缩小关键词"}
    # 4. 审计日志
    audit_log(args, result, ctx.trace_id)
    # 5. 返回 LLM 友好格式(精简,不裸 SQL/不内部字段)
    return {"matched_rules": [{"id": r.id, "text": r.summary} for r in result[:5]]}

3.4 易踩坑

  1. 把 SQL 拼接交给 LLM → SQL 注入。Tool 永远只接领域名词,内部翻译成 SQL。
  2. 工具结果裸返回——返回 5MB JSON,把 context 撑爆。要在 Tool 内部做 summarize/topK 截断。
  3. 不写 timeout——单个工具卡 30s,LLM 看不到结果一直等。所有 Tool 强制 timeout(默认 5s,可配置)。
  4. 不做幂等——重试机制让「下单」工具调用两次。写操作工具必须带 idempotency_key。
  5. 工具数量超过 20 个——LLM 选错工具概率飙升。要做工具路由(先选工具集,再在子集里精选)。

4死循环防控 6 招

4.1 五句话原理

  1. Agent 死循环是头号生产事故:同一 Action 反复调、Thought 自我说服、永远到不了 Final Answer。
  2. 根因三类:① 工具结果不变(LLM 看到一样的 obs 做一样的判断);② Prompt 没明示终止条件;③ LLM 抖动(同样输入不同步骤间答案飘)。
  3. 核心防御:步数上限 + 重复检测 + 强制收敛三件套。
  4. 高级:Self-Critique(每 N 步让 LLM 自评进度,无进展就收敛)+ 外部监督员(独立 LLM 看轨迹,认定卡住就强制停)。
  5. 2026 行业事实:没有任何 Agent 能 100% 不卡,关键是检测 + 优雅降级 + 人工兜底三层网。

4.2 6 招防御机制

招式原理触发条件动作
① 硬步数上限循环计数step >= 10强制结束,返回当前轨迹
② 重复 Action 检测记录 (tool, args) hash同 hash 连续 2 次提示 LLM「不要重复」或强制结束
③ 进度自评每 3 步问 LLM「有进展吗」连续 2 次「无进展」触发 Re-plan 或人工
④ 时间预算累计 elapsed 时间>30s强制结束,返回部分结果
⑤ token 预算累计 prompt+completion token>50k强制结束
⑥ 监督员模型另一个 LLM 看轨迹判定「卡住」外部中断当前 Agent

4.3 代码:重复检测 + 步数上限

import hashlib

class AgentGuard:
    def __init__(self, max_steps=10, max_token=50000, max_time=30):
        self.max_steps = max_steps
        self.action_history = []     # 最近 N 个 (tool, args_hash)
        self.start = time.time()
        self.token_used = 0

    def should_stop(self, step, last_action) -> tuple[bool, str]:
        if step >= self.max_steps:
            return True, "超过最大步数"
        if time.time() - self.start > 30:
            return True, "超时"
        if self.token_used > 50000:
            return True, "超出 token 预算"
        # 重复检测:连续 2 次相同 (tool, args)
        h = hashlib.md5(str(last_action).encode()).hexdigest()
        self.action_history.append(h)
        if len(self.action_history) >= 2 and self.action_history[-1] == self.action_history[-2]:
            return True, "重复 Action,疑似死循环"
        return False, ""

4.4 易踩坑

  1. 只设步数不设时间 → 单步 30s 卡满 10 步 = 5 分钟,客户端早超时。要步数 + 时间双兜底。
  2. 重复检测只看 tool 名 → 「search('猫')」和「search('狗')」算重复。要(tool, args) 整体 hash。
  3. 触发上限直接抛异常 → 用户体验差。要返回部分结果 + 兜底文案(「我已尝试多次但未能完成,以下是已收集到的信息...」)。
  4. 不上报死循环事件 → 看不到趋势。每次触发要Trace 上报,日报统计哪类任务最容易卡,反推 Agent 设计优化。

5工具失败降级 + Schema 校验

5.1 五句话原理

  1. 线上工具失败是常态不是异常:超时、限流、第三方挂、参数错、权限不足——必须把每种失败的处理路径写明。
  2. 降级层级:① 本工具重试 1-2 次(瞬时故障);② 换备用工具(同功能不同后端);③ 降级返回(返回缓存或简化结果);④ 告知 LLM 工具不可用,让它选别的;⑤ 升级到人工。
  3. Schema 校验:LLM 输出的工具参数必须先校验(Pydantic / JSON Schema),通过才执行,失败把错误信息回灌给 LLM 让它修(称为 Self-Correction)。
  4. 错误回灌格式很关键:不要回 stack trace,要回结构化、人话化错误描述+ 修复建议。
  5. 2026 主流模型对错误回灌的修复成功率(实测内部审核场景):GPT-5 / Claude 4.5 Sonnet ~92%、Qwen3-Max ~88%、DeepSeek-V3 ~85%——错误回灌是 Agent 自愈的核心机制。

from pydantic import ValidationError

def execute_with_correction(llm, tool, llm_output, max_retry=2):
    """LLM 调用工具,失败把错回灌让它修"""
    for attempt in range(max_retry):
        try:
            args = tool.args_schema(**llm_output["arguments"])
            return tool.run(args)
        except ValidationError as e:
            # 把人话错误回灌给 LLM
            error_msg = {"error": "参数校验失败", "details": e.errors(), "fix_hint": "请按 schema 修正参数后重试"}
            llm_output = llm.invoke([
                {"role": "system", "content": "上次调用失败,请修正:"},
                {"role": "user", "content": str(error_msg)}
            ])
        except TimeoutError:
            return {"error": "工具超时", "hint": "请简化请求或换工具"}
    return {"error": "重试用尽", "hint": "请改变策略或上报人工"}

  4. 降级路径未演练——线上突然第三方挂了才发现降级根本没走通。要做故障注入演练(Chaos Engineering)。

6记忆三层架构 · Short / Long / Episodic

6.1 五句话原理

  1. Agent 没记忆 = 金鱼,每次对话都从头开始。生产 Agent 必须有三层记忆。
  2. Short-term(短期):当前会话/任务内的对话历史,放在 messages 里,有 trim 策略防止 context 爆炸。
  3. Long-term(长期):跨会话的用户偏好/事实/历史决策,通常存数据库或向量库,需要时检索注入。
  4. Episodic(情景):过去的成功/失败案例(query+轨迹+结果),用于 Few-shot 启发或避免重蹈覆辙——本质是 Agent 自己的「经验库」。
  5. 2026 趋势:Memory 不再是「拼接进 Prompt」,而是独立的 Memory Service(LangGraph Store、mem0、letta),有 CRUD、检索、TTL、隐私边界。

Short-term当前对话/任务消息messages 数组会话期0-50 条
Long-term用户事实/偏好/ProfileSQL + 向量库永久或长期100-10000 条

# 短期:trim by token,保留 system + 最近 N 轮
def trim_short_term(messages, max_token=8000):
    system = messages[0]
    recent = messages[-1:]   # 最新一条必留
    middle = messages[1:-1]
    # 倒序累加,超 max_token 就停
    used, kept = count_token([system] + recent), []
    for m in reversed(middle):
        t = count_token([m])
        if used + t > max_token: break
        kept.insert(0, m); used += t
    return [system] + kept + recent

  1. 无脑全塞短期——历史 50 轮全堆 Prompt,LITM 严重 + 成本飙。要 trim 或 summarize 旧轮。
  2. 长期记忆不分类——所有事实混一锅,检索噪声大。要按类别(profile/preference/history)分桶。
  3. Episodic 不区分成败——LLM 也会学失败案例。检索时默认 only_success=True,失败案例放另一个库做反例学习。
  4. Memory 没有 TTL——用户三年前说「我喜欢咖啡」一直影响今天决策。关键事实必须带 timestamp 和 confidence。

第 7 章:Context Engineering——2026 最新工程范式

7.1 五句话原理

① Context Engineering ≠ Prompt Engineering。Prompt 是「写一句好指令」,Context 是「设计 Agent 每一步看到的整个上下文窗口」——包括 System、历史、工具结果、记忆、外部知识。
② 核心命题:Token 是有限资源。32K/128K 看似很大,塞 10 轮工具调用 + RAG 50 段就爆了。Context Engineering 就是在每一步动态决定:留什么、删什么、怎么排。
③ 五大组件:System Prompt(身份+边界)、Working Memory(最近 N 轮)、Retrieved Knowledge(RAG 结果)、Tool Results(工具返回)、Long-term Memory(事实/偏好)。
④ 三大动作:Select(从所有来源选什么)、Compress(摘要/裁剪)、Order(顺序按重要性排,关键的放最后防 LITM)。
⑤ 为什么 2026 突然火?Agent 多步执行后,中间产物爆炸增长。「写好 prompt」的红利吃完了,下一波是「写好 context 调度器」。

7.3 完整代码:Context 调度器

class ContextEngine:
    """统一管理一次 LLM 调用前的上下文组装"""
    def __init__(self, max_token=12000):
        self.max_token = max_token
        self.system = ""
        self.working = []       # [{"role","content"}, ...]
        self.retrieved = []     # [{"text","score","source"}, ...]
        self.tool_results = []  # [{"tool","output","ts"}, ...]
        self.long_term = []     # 已检索的相关记忆

    def select(self, user_query):
        """① Select:决定每个来源选哪些"""
        # RAG 只留 Top3,Tool 结果只留最近 5 次
        self.retrieved = sorted(self.retrieved, key=lambda x: -x["score"])[:3]
        self.tool_results = self.tool_results[-5:]
        # 长期记忆按 query 相关性过滤
        self.long_term = [m for m in self.long_term if related(m, user_query)]

    def compress(self):
        """② Compress:超长内容压缩"""
        for r in self.tool_results:
            if count_token(r["output"]) > 800:
                r["output"] = summarize(r["output"], max_token=300)
        # Working Memory 滑窗+摘要
        if count_token(self.working) > 6000:
            old = self.working[1:-4]   # 保留 system + 最近 4 轮
            summary = summarize_messages(old)
            self.working = [self.working[0],
                            {"role":"system","content":f"[历史摘要] {summary}"}
                            ] + self.working[-4:]

    def order(self):
        """③ Order:按 LITM 原则排,关键的放最后"""
        return [
            {"role":"system","content": self.system},
            *self.long_term_to_msgs(),       # 弱相关放中间
            *self.retrieved_to_msgs(),
            *self.working,                   # 最重要放最后
            *self.tool_results_to_msgs(),
        ]

    def build(self, user_query):
        self.select(user_query)
        self.compress()
        msgs = self.order()
        # 兜底:总长度仍超限就再裁
        while count_token(msgs) > self.max_token:
            msgs.pop(2)   # 弹出 retrieved 中最弱的
        return msgs

7.4 易踩坑

坑 1:工具结果不压缩。调一次搜索 API 返回 5KB JSON,5 轮就 25KB——Context 直接爆。必须有 summarize 兜底。
坑 2:System Prompt 里塞太多。System 应该是「身份 + 边界 + 工具列表」,不要把 RAG 知识塞进去——那是 Retrieved 的活。
坑 3:历史摘要太频繁。每轮都 summarize 会导致信息丢失累积。设阈值(超过 6K 才压),不要无脑压。
坑 4:不区分 Working 和 Long-term。对话里的「我叫张三」是 Working,用户偏好「不喜欢长回复」是 Long-term。混在一起 trim 时会丢关键信息。
坑 5:LITM 摆放错。关键信息(用户最新问题、最近工具结果)必须放最后,不要放在历史摘要后面。

第 8 章:Workflow 约束 Agent + Orchestrator-Worker 多智能体

8.1 五句话原理

① 一句话核心:Agent 越自由越不稳。Anthropic 2024 论文《Building Effective Agents》核心观点——能用 Workflow 解决的就别用 Agent。
② Workflow vs Agent 边界:Workflow 是预定义路径(if-else/状态机),路径少且可预测;Agent 是LLM 自主决策路径,路径不固定。
③ 五种 Workflow 模式:Prompt Chaining(串行)、Routing(分类后分流)、Parallelization(并行)、Orchestrator-Workers(主管派活)、Evaluator-Optimizer(生成-评价循环)。
④ Orchestrator-Worker 是审核场景的最佳形态:主 Agent 拆解任务 → 多个 Worker(文本审核/图片审核/规则匹配)并行 → 主 Agent 汇总判决。
⑤ 落地原则:能用 Workflow 替代 Agent 的部分用 Workflow;Agent 只用在「需要动态决策的少数节点」。这样既稳定又灵活。

class ModerationOrchestrator:
    """审核主管:拆解 → 派 Worker → 汇总"""
    def __init__(self, workers):
        # workers = {"text": TextModWorker, "image": ImageModWorker, "rule": RuleWorker}
        self.workers = workers

    def plan(self, content):
        """让 LLM 决定要派哪些 Worker"""
        prompt = f"""内容:{content}
        判断需要哪些审核 Worker(可多选):
        - text:文本审核(涉黄涉政)
        - image:图片审核(暴恐违规)
        - rule:规则引擎(关键词命中)
        输出 JSON: {{"workers":["text","image"]}}"""
        return json.loads(llm.call(prompt))["workers"]

    def execute(self, content):
        # ① 主 Agent 拆解
        selected = self.plan(content)
        # ② 并行派 Worker
        results = {}
        with ThreadPool(len(selected)) as pool:
            futures = {w: pool.submit(self.workers[w].run, content)
                       for w in selected}
            for w, f in futures.items():
                results[w] = f.result()
        # ③ 主 Agent 汇总判决
        return self.aggregate(content, results)

    def aggregate(self, content, results):
        """主 Agent 看所有 Worker 结果做最终判决"""
        prompt = f"""内容:{content}
        各 Worker 结果:{json.dumps(results, ensure_ascii=False)}
        最终判决(放出/上报/屏蔽)+ 理由。
        输出 JSON: {{"verdict":"...","reason":"...","confidence":0.x}}"""
        return json.loads(llm.call(prompt))

第 9 章:Reflection 自纠 + Multi-Agent 协作

9.1 五句话原理

① Reflection 是 Agent 自我纠错的机制:生成一个答案后,让 LLM 自己评价(或让另一个 LLM 评价),发现问题就重新生成。
② 三种 Reflection 形态:Self-Critique(同一个 LLM 自评)、Generator-Critic(两个 LLM 角色分离)、Reflexion(把失败原因写进 Memory 下次复用)。
③ 关键不是「让它反思」,是「反思后能改进」。Critic 必须输出结构化错误清单(哪一句错、错在哪、怎么改),Generator 才能针对性修复。
④ Multi-Agent 不等于多 Agent。真正的 Multi-Agent 是角色分工明确、有协作协议(谁先说、谁拍板、谁兜底),否则就是「N 个 Agent 抢话」。
⑤ 不要为了酷而上 Multi-Agent。2 个 Agent 能解决的不要上 5 个——多一个 Agent 多一份失败概率,且 token 成本翻倍。

9.2 Reflection 三种形态对照

Self-Critique同 LLM 先生成再自评简单纠错(语法/格式)2x 调用
Generator-Critic两个不同角色 LLM需要多视角(写作/代码)2x 调用
Reflexion失败案例写入 Memory多轮任务(长期任务)更省(下次直接学到)

class ModerationGeneratorCritic:
    """生成-评价循环:提升审核理由质量"""
    MAX_ROUND = 3

    def generate(self, content, prev_critique=None):
        prompt = f"""内容:{content}
        给出审核判决和理由(放出/上报/屏蔽)。"""
        if prev_critique:
            prompt += f"\n上一版反馈:{prev_critique}\n请针对性修正。"
        return llm_generator.call(prompt)

    def critique(self, content, verdict):
        """Critic 输出结构化问题清单,而不是泛泛点评"""
        prompt = f"""内容:{content}
        判决:{verdict}
        作为审核质检员,从 4 个维度评价(每条 0-5 分 + 问题点):
        1. 判决是否准确(是否漏判/误判)
        2. 理由是否引用具体文本
        3. 是否提到适用规则
        4. 用词是否专业(避免主观词)
        输出 JSON: {{"scores":{{...}},"issues":["..."],"pass":true/false}}"""
        return json.loads(llm_critic.call(prompt))

    def run(self, content):
        verdict = self.generate(content)
        for i in range(self.MAX_ROUND):
            review = self.critique(content, verdict)
            if review["pass"]:
                return verdict, i+1
            verdict = self.generate(content, review["issues"])
        return verdict, self.MAX_ROUND  # 兜底返回最后一版

第 10 章:MCP 协议——Anthropic 推的 Agent 工具标准

10.1 五句话原理

① MCP(Model Context Protocol)是 Anthropic 2024 年底推的「Agent 工具协议」:统一 LLM 调用外部工具/资源的接口规范,目标是「USB-C for AI」。
② 解决的痛点:每个 Agent 框架(LangChain/LlamaIndex/Cursor/Cline)都自己定义工具接口,工具开发者要适配 N 种格式。MCP 出现后,写一次工具,所有支持 MCP 的客户端都能用。
③ 三大核心概念:Resources(只读资源,如文件/数据库)、Tools(可执行函数)、Prompts(预定义模板)。客户端 Host(LLM 端)通过 Server(工具端)调用三类能力。
④ 通信形态:Stdio(本地进程通信,最常用)/ HTTP+SSE(远程服务)。JSON-RPC 协议格式。
⑤ 为什么要懂 MCP?2026 年 Cursor、Cline、Claude Desktop、Codex 全部支持 MCP——内部工具一旦暴露成 MCP Server,所有工具客户端通用,工具复用率拉满。

概念定位类比 Spring典型例子

Tools可执行函数(可写)@PostMapping发邮件 / 创建工单

第 11 章:权限治理 + 可观测——Agent 上线前必过的两道关

11.1 五句话原理

① Agent 自由度越高,权限风险越大。能调工具的 Agent = 能改数据库的程序员,必须有权限模型。
② 权限三层模型:身份(Who)、动作(What)、资源(On What)。Agent 调用任何 Tool 都要带用户身份,Tool 内部按用户权限校验。
③ 危险操作需要 Human-in-the-Loop:删除/支付/外发邮件 这类不可逆操作,Agent 不能独立完成,必须返回「确认请求」给人类拍板。
④ 可观测三件套:Logging(每步输入输出) + Tracing(调用链路) + Metrics(成功率/延迟/成本)。没这三个 Agent 出了问题完全黑盒。
⑤ Trace 不是日志而是结构化:每次 LLM 调用、每次 Tool 调用都是一个 Span,有 parent_id 串成树。LangSmith / Langfuse / Phoenix 是主流。

from opentelemetry import trace
tracer = trace.get_tracer("agent")

class GovernedAgent:
    HIGH_RISK_TOOLS = {"delete_record","send_email","pay","update_status"}

    def __init__(self, user_id, role):
        self.user_id = user_id
        self.role = role
        self.audit_log = []

    @tracer.start_as_current_span("agent_step")
    def step(self, plan):
        tool, args = plan["tool"], plan["args"]

        # ① 权限校验
        if not has_permission(self.role, tool, args):
            raise PermissionDenied(f"{self.role} 不能调用 {tool}")

        # ② HITL:高危操作返回确认请求
        if tool in self.HIGH_RISK_TOOLS:
            confirm_id = save_pending(self.user_id, tool, args)
            return {"status":"need_confirm",
                    "confirm_id": confirm_id,
                    "preview": render_preview(tool, args)}

        # ③ 实际调用 + Trace + Audit
        with tracer.start_as_current_span(f"tool.{tool}") as span:
            span.set_attribute("user_id", self.user_id)
            span.set_attribute("tool", tool)
            span.set_attribute("args", json.dumps(args))
            result = call_tool(tool, args)
            span.set_attribute("result_length", len(str(result)))

        self.audit_log.append({
            "ts": now(), "user": self.user_id, "tool": tool,
            "args": args, "result_summary": summary(result)
        })
        return {"status":"ok","result":result}
