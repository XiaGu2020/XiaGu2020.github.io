⑤ 装饰器（@Decorator，本质是函数包装）

import time, functools

def timer(fn):
    @functools.wraps(fn)
    def wrap(*args, **kwargs):
        t = time.time()
        r = fn(*args, **kwargs)
        print(f"{fn.__name__} 用时 {time.time()-t:.3f}s")
        return r
    return wrap

@timer
def slow():
    time.sleep(1)

slow()  # slow 用时 1.001s
对照：装饰器约等于 Java 的 AOP 切面，但是函数级、运行时、可组合。@app.route / @pytest.fixture 都是它。

2.5 GIL 必须知道

⚠️ 面试必问：GIL（Global Interpreter Lock）= 一个进程同一时刻只有一个线程执行 Python 字节码。
影响：多线程做 CPU 密集任务无并行加速，得用多进程或 asyncio。
但 IO 密集（网络/磁盘）多线程仍有效（IO 阻塞时释放 GIL）。
结论：调大模型 API 这种 IO 密集场景，用 asyncio + httpx 或 concurrent.futures.ThreadPoolExecutor 都行。

6流式输出（必会，前端体验关键）

为什么要流式？

非流式：等模型生成完才返回，用户盯着白屏等 5 秒。
流式（SSE）：每生成一个 token 就推一段，用户立即看到打字效果，TTFT (Time to First Token) 大幅降低。

stream = client.chat.completions.create(
    model="qwen-plus",
    messages=[{"role":"user","content":"写一首关于审核的打油诗"}],
    stream=True,
)

for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        print(delta.content, end="", flush=True)
print()
面试连接：什么是 SSE？答：Server-Sent Events，HTTP 之上的服务器单向推送协议，Content-Type: text/event-stream，每条以 data: 开头、\n\n 结尾。OpenAI 流式就是 SSE。
7Function Calling（让大模型会用工具）

7.1 核心理解：模型不真调函数，它只输出"调用意图"

流程：① 你给模型工具定义 → ② 模型决定要调哪个工具+参数 → ③ 你的代码实际执行 → ④ 把结果回填给模型 → ⑤ 模型生成最终回答。

import json

# ① 定义工具（JSON Schema 风格）
tools = [{
    "type": "function",
    "function": {
        "name": "check_sensitive_word",
        "description": "检测一段文本中是否包含敏感词",
        "parameters": {
            "type": "object",
            "properties": {
                "text": {"type": "string", "description": "待检测文本"}
            },
            "required": ["text"]
        }
    }
}]

# 真正的工具实现（你写）
def check_sensitive_word(text: str):
    bad = ["违禁词1", "违禁词2"]
    hits = [w for w in bad if w in text]
    return {"hit": bool(hits), "words": hits}

# ② 第一次调用：模型可能返回 tool_calls
messages = [{"role":"user", "content":"帮我检查这句话有没有问题：xxx 内容 yyy"}]
resp = client.chat.completions.create(
    model="qwen-plus", messages=messages, tools=tools,
)
msg = resp.choices[0].message

if msg.tool_calls:
    messages.append(msg)            # 把模型的 assistant 消息原样回填
    for call in msg.tool_calls:
        args = json.loads(call.function.arguments)
        result = check_sensitive_word(**args)
        # ③ 把工具结果作为 tool 角色喂回去
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": json.dumps(result, ensure_ascii=False)
        })

    # ④ 第二次调用：模型基于工具结果生成最终回答
    final = client.chat.completions.create(model="qwen-plus", messages=messages)
    print(final.choices[0].message.content)
⚠️ 三个易错点
  1. 第一次调用后必须把 msg（含 tool_calls）原样 append，不能丢
  2. tool 角色消息的 tool_call_id 必须对应模型给的 id
  3. 有些场景模型会在一次回复里要求并行调多个工具，要循环处理

10第一个 mini-RAG（最重要的里程碑）

10.1 RAG 是什么（一段话原理）

RAG = Retrieval-Augmented Generation。当你的私有知识没在大模型训练数据里时，先把它切片+向量化存到向量库，提问时用语义检索找相关片段，把片段+问题一起喂给大模型生成回答。核心：检索给大模型补充"上下文"。

10.2 最小 RAG（70 行代码搞定）

from openai import OpenAI
from sentence_transformers import SentenceTransformer
import chromadb
from dotenv import load_dotenv
import os

load_dotenv()

# ① 你的私有知识（实际场景从文件读）
docs = [
    "阅文集团是中国最大的网络文学平台。",
    "网文审核需关注涉黄、涉政、暴力等敏感内容。",
    "Qwen3Guard 是阿里推出的内容安全审核模型。",
    "漫剧审核中台支持新剧首发审核与连载更新审核。",
]

# ② Embedding 模型（中文优先 BGE）
embedder = SentenceTransformer("BAAI/bge-small-zh-v1.5")

# ③ 向量库（Chroma 本地存）
chroma = chromadb.Client()
col = chroma.create_collection("kb")
col.add(
    documents=docs,
    embeddings=embedder.encode(docs).tolist(),
    ids=[f"doc{i}" for i in range(len(docs))],
)

# ④ 检索 Top-K
def retrieve(query, k=2):
    q_vec = embedder.encode([query]).tolist()
    res = col.query(query_embeddings=q_vec, n_results=k)
    return res["documents"][0]

# ⑤ 拼 Prompt 调大模型
client = OpenAI(
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
)

def rag(query):
    contexts = retrieve(query)
    prompt = f"""
请基于以下资料回答问题，资料中没有则回答"未在资料中找到"。

资料：
{chr(10).join(f"- {c}" for c in contexts)}

问题：{query}"""
    resp = client.chat.completions.create(
        model="qwen-plus",
        messages=[{"role":"user","content":prompt}],
        temperature=0.1,
    )
    return resp.choices[0].message.content

print(rag("漫剧审核中台支持哪些场景？"))

10.3 RAG 全流程图（5 步死记）

用户提问 ─┐
          ▼
   [Embedding 编码] ──→ 查询向量
                          │
                          ▼
   [向量库检索 TopK] ──→ 相关片段
                          │
                          ▼
   [Prompt 组装]   ──→ 资料+问题
                          │
                          ▼
        [LLM 生成]    ──→ 最终回答

🎯 Phase 0 高频面试题（25 道，含答案要点）

1. Python 和 Java 主要差异？
动态 vs 静态类型；缩进 vs 大括号；GIL（全局解释器锁，多线程 CPU 不并行）；Python 函数是一等公民；没有真正的 private；解释执行；包管理通过 pip/uv 而非 maven。AI 生态主要在 Python，Java 通过 Spring AI/LangChain4j 集成。
2. 什么是 GIL？对什么场景有影响？
Global Interpreter Lock，CPython 中保证同一时刻只有一个线程执行字节码。CPU 密集多线程无加速（用多进程或 numpy 等 C 扩展释放 GIL）；IO 密集（网络/磁盘）多线程仍有效。AI 调 API 是 IO 密集，asyncio 或线程池都行。
3. 列表推导式和普通 for 哪个快？
推导式略快（C 层优化），但更主要是可读性。复杂逻辑还是用 for 更清晰。
4. 什么是装饰器？举一个 AI 场景的用法
本质是接收函数返回函数的高阶函数，用 @ 语法糖。AI 场景：@retry(3) 重试 LLM 调用、@cache 缓存 embedding、@trace 记录 token 用量。
5. yield 和 return 区别？流式输出怎么实现？
return 一次性返回；yield 暂停函数返回值，下次从这里继续，函数变成 generator。流式输出本质就是 generator + SSE 协议，每生成一段 yield/推送一段。
6. NumPy 为什么比 Python list 快？
① 连续内存（cache 友好）② C 实现的向量化运算 ③ 类型统一无需逐元素类型检查 ④ 释放 GIL 可并行。差距通常 10–100 倍。
7. NumPy 广播规则？
两个数组从右向左对齐维度，每一维要么相等要么其中一方为 1 或缺失才能广播。例：(3,4)+(4,) OK；(3,4)+(3,1) OK；(3,4)+(2,4) 报错。
8. Pandas 的 apply / map / applymap 区别？
map 用于 Series 元素级；apply 用于 Series 或 DataFrame，可作用行/列；applymap（已弃用，改用 DataFrame.map）作用 DataFrame 每个元素。性能：向量化 > apply > iterrows。
9. 怎么处理 Pandas 大文件？
① chunksize 分批读 ② 指定 dtype 节省内存 ③ 只读需要列 usecols ④ 用 Parquet 替代 CSV ⑤ 上 polars/duckdb 替代。
10. 大模型 API 的 messages 数组是什么作用？
多轮对话上下文，按时间顺序拼接 system/user/assistant/tool 消息。每次调用都要把历史发回去（模型本身无状态）。多轮对话需自己管理上下文长度（超过窗口要裁剪/摘要）。
11. system / user / assistant / tool 各自意义？
system 设定角色与规则；user 用户输入；assistant 模型上一轮输出（多轮回填）；tool 工具调用结果（Function Calling 第二阶段回填）。
12. temperature 和 top_p 区别？
temperature 调采样分布的"陡峭度"，越高越随机；top_p 核采样，从累计概率达 p 的最小集合里采。两个一般二选一调，默认调 temperature 即可。分类用 0–0.3，创作 0.7+。
13. 什么是 SSE？流式响应怎么实现？
Server-Sent Events，HTTP 长连接服务器单向推送，Content-Type: text/event-stream，每条 data: 开头 \n\n 结尾。OpenAI 流式就是 SSE，客户端循环解析每个 chunk 拿 delta.content 拼接。
14. Function Calling 原理？模型真的执行函数吗？
不真执行。模型根据工具 schema 决定要调哪个工具+参数，输出 tool_calls；客户端代码实际执行后把结果以 tool 角色回填，模型再生成最终回答。本质是结构化输出+多轮对话。
15. Function Calling 与 Agent 关系？
Function Calling 是 Agent 的基础能力。Agent 在此之上加了循环（多步执行）+ 自主决策（下一步做什么）+ 反馈（观察环境）。单次调工具不算 Agent，循环+决策才算。
16. JSON Mode 和 Function Calling 区别？
JSON Mode 只保证输出是合法 JSON，结构靠 Prompt 约束；Function Calling 提供严格 schema，模型按 schema 输出 + 触发后续工具调用流程。要稳定结构化结果用 response_format/parse；要工具调用用 Function Calling。
17. 什么是 Tokenizer？为什么要 Token？
Tokenizer 把文本切成 token（介于词和字符之间的子词单元，常用 BPE）。模型只懂 token 不懂字符。中文 1 个汉字 ≈ 1.5–2 token，英文 1 单词 ≈ 1.3 token。Token 决定上下文窗口和计费。
18. 上下文窗口（Context Window）超了怎么办？
① 截断早期消息 ② 摘要旧轮次 ③ RAG 把长知识抽离到向量库按需检索 ④ 用更大窗口模型（Qwen 128K）⑤ 多轮拆任务。最优方案视场景。
19. 什么是 RAG？最小流程？
Retrieval-Augmented Generation。流程：文档切块 → embedding → 向量库 → 查询时用问题向量召回 TopK → 把片段+问题拼 Prompt → LLM 生成。解决：私有知识不在训练集、知识更新快、可溯源。
20. 什么时候用 RAG，什么时候用微调？
RAG：知识更新频繁、可溯源、量大；微调：风格/格式/领域语言固化、需要离线推理、私有部署。两者经常组合：RAG 注入知识 + 微调注入风格。
21. Embedding 是什么？中文常用什么模型？
把文本映射到固定维度向量（如 1024 维），语义相近向量距离近。中文常用：BGE-M3（多语种）、BGE-large-zh-v1.5、M3E、text-embedding-v3（阿里）。审核场景考虑领域微调。
22. 向量数据库存什么？怎么选？
存向量+原始文本+元数据。轻量 demo 用 Chroma；中型生产用 Qdrant/Milvus；已有 PG 用 pgvector；阿里云上用 DashVector。索引算法 HNSW 兼顾召回与延迟。
23. Spring AI 和 LangChain4j 选哪个？
已用 Spring Boot 选 Spring AI（无缝集成 Bean/Profile/AOP/Actuator）；非 Spring 项目或喜欢 LangChain 风格选 LangChain4j（抽象更多，社区组件多）。两者都支持流式/Function/RAG/Memory。
24. Java 调大模型注意什么？
① 超时设大（30s+，流式更长）② 重试+幂等 ③ 限流（用 Bucket4j 或 Redis）④ 流式用 Flux/SseEmitter ⑤ 监控 token 用量+成本 ⑥ 异步调用避免阻塞 web 线程。
25. 怎么省 API 成本？
① 选合适模型（90% 场景 mini 级够用）② 缓存（同 Prompt 直接返回）③ Prompt 缓存（OpenAI/Anthropic 支持，长 system 大幅省钱）④ max_tokens 限制 ⑤ Batch API（半价但慢）⑥ 路由（简单问题走小模型）⑦ RAG 替代长上下文。
